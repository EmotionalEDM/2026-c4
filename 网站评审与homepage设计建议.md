## 一、必修 bug：前后端文案 key 不匹配

前端引用了若干 `ui_text.py` 里**不存在**的 key。`textContent = undefined` 会把字面
**"undefined"** 渲染到页面上——评委一眼就能看到。全部只需在 `backend/app/ui_text.py` 补 key
（或删前端死代码），前端零改动。

| # | 现象 | 前端引用处 | 后端缺失的 key | 严重度 |
|---|------|-----------|---------------|--------|
| 1 | 顶部导航品牌副标题显示 **"undefined"** | app.js:129 `m.brand_sub` | `meta.brand_sub` | 🔴 高（每页可见） |
| 2 | 「② 选择展示案例」标题下方显示 **"undefined"** | app.js:156 `S.cases.desc` | `sections.cases.desc` | 🔴 高 |
| 3 | **「真实数据」标签永远渲染成一个空药丸** | app.js:394 `UI.src_tag.real` | `src_tag.real`（现只有 `sim`） | 🔴 高——这恰恰是全站最想强调的卖点，目前真实数据反而没有标签，simulated 才有 |
| 4 | 选中案例后的「当前案例」行丢失模型名与真实数据标识 | app.js:226-230 用到 `UI.story.src_real`，且 `story.cur_case` 模板里没有 `{src_tag}`/`{model}`/`{agent}` 占位符 | `story.src_real` + 模板占位符 | 🟡 中（功能写了却显示不出来） |

## 二、语言与逻辑表述把关（通读结论）

整体文风是好的：浅显不白话、术语前后基本一致（SkillIR / AGC 胶囊 / CV-ECC、True-WS/False-WS/Margin、
trace_xxx=0 / xxx_trace=1 的规则在 narratives / content / ui_text 三处我交叉核对过，**一致**）。
以下是真正值得改的点，按严重度排序。

### 2.1 「模拟演示」三个字在打自己的脸 🔴

全站最核心的卖点是「真实数据、真实模型输出、与论文指标交叉印证」，但：

- `sections.story.h2` = "③ **模拟**演示"（ui_text.py:86）
- `meta.title` = "…买家溯源**模拟**演示"（ui_text.py:22）

评委看到"模拟"二字，对后面所有 `real` 标签的信任会先打折。建议改为
「③ 全流程演示」或「③ 案例实录」，title 同理（"全流程演示平台"）。
（attacks.py 里确实存在 simulated 回退路径，但那是兜底，不该写进门面标题。）

### 2.2 ①~⑤ 编号跨视图断裂 🔴

页面已拆成 首页 / 演示 / 指标 三个 hash 视图，但区块编号还是单页时代的 ①~⑤：
评委点进「演示」视图，看到的第一个标题是「**②** 选择展示案例」——①在哪？
「⑤ 实验指标总览」单独成页，前面没有①~④。

建议：**去掉编号**，或每个视图内重新从 1 编号（演示视图：1 选案例 → 2 故事流 → 3 结果裁决）。
编号本来是引导单页滚动的，现在路由已经承担了引导职责。

### 2.3 「判决」一词三个写法 🟡

- `sections.verdict.h2` = "④ 结果**判断**"
- `verdict.title` = "📋 **判决书** · {title}"
- README / 修改计划 = "④ 结果**裁决**"

建议统一为「结果裁决」（区块标题），卡片标题保留「判决书」是好的拟物，但"结果判断"最弱，删掉。

### 2.4 超长平台名压垮三个位置 🟡

「面向大模型Agent的结构化技能版权审计与溯源平台」（22 字）出现在：

1. **导航品牌位**（15px 单行，brand_title）——窄屏必然溢出或换行；
2. **指标对比表列头**（ui_text.py:294 `compare_headers[1]`）——一列表头 22 个字，
   表格布局会被严重挤压，旁边 PromptCARE/PromptCOS 只有 9~10 字符，视觉极不平衡；
3. **footer**（ui_text.py:117）——"……溯源平台比赛展示系统"两个名词连写，读着像没断句。

建议：导航主标题用短名 **SkillCODER**，副标题（brand_sub，正好是 bug#1 要补的 key）放全名；
表头第二列改成「**本方法**」或「SkillCODER（本文）」；footer 加间隔符：
"面向大模型Agent的结构化技能版权审计与溯源平台 · 比赛展示系统"。

### 2.5 「水印成本降低约15倍」是语病 🟡

中文里"降低"不能搭配"倍"（hero 实验验证卡，ui_text.py:67）。改为：

- "水印开销仅为基线的 **1/15**"，或
- "水印成本降低约 **93%**"。

### 2.6 保真度数字与文案自相矛盾 🔴（逻辑问题）

指标页保真度板块（metrics.py:43-52）：

> 「效用下降 Utility Drop（**越低越好**）：本方法 **0.662** vs PromptCARE 0.700」
> takeaway：「加水印后普通任务表现**几乎无损**」

字面读下来是"效用下降了 66.2%"，却说"几乎无损"——评委一定会抓住这个问。
请核对论文口径：这个 0.662 究竟是什么尺度（归一化分数？某 loss？还是其实是"效用保持率"？）。
无论哪种，都要在指标名或表注里写明尺度，比如改名为论文里的原始指标名并加一行
"注：该指标为…，数值含义是…"。**这是全站唯一一处我认为评委必然质疑的逻辑硬伤。**

### 2.7 两类攻击 vs 四类攻击的对应关系没人讲 🟡（逻辑问题）

- 案例台讲「第一类（盲攻击）/ 第二类（定向删除）」两类；
- 指标页鲁棒性表却是「改写 / 压缩 / 辅助条款删除 / 章节重排」四类；
- narratives 第 5 步 technical_hint 也提"四类攻击"。

两套分类都对（第一类 = 改写/压缩/重排三种盲攻击的统称，第二类 = 辅助条款删除），但**全站没有
一句话把它们对应起来**。建议在攻击图例（`sections.cases.attack_legend`）或鲁棒性表的 takeaway
里补一句："第一类对应改写/压缩/重排三种盲攻击，第二类对应辅助条款删除。"

### 2.8 小的措辞打磨 🟢

| 位置 | 现状 | 建议 |
|------|------|------|
| hero.lead（ui_text.py:52） | "现有水印方法**失效**" | 过于绝对，且与三卡里"现有水印只能判定是否被盗、无法追踪谁泄露"的准确表述矛盾。统一为后者的说法 |
| hero.h1 | "证明并**追凶**" | 风格化表达，演示场合可以保留；若评委偏严肃学术，可备选"证明所有权、追出泄露者"。建议和队长定一个基调 |
| score_bar 阈值标签（app.js:597） | "判定阈值 τ_o = **50%**" | 公式里 Score_own 是 0~1 的分数，τ_o 显示成百分比口径混乱，建议显示 `0.5` |
| 第 5 步标题（narratives.py:66） | "攻击过程" | 与 README 的"攻击者复制 / 改写"不一致，"攻击者改写 / 重新部署"更具体 |
| hero.lead | "数千美元的数字资产" | 国内比赛场合可考虑"上千美元（数万元）级"或干脆"高价值数字资产"，避免币种出戏 |

### 2.9 通读后没有问题的部分（可以放心）

- 八步 story 文案逻辑链完整：每步"上一步结果 → 本步动作 → 为什么"都讲清了；
- 第 6 步"激活-沉默非对称性"的解释（narratives + content 两处）是全站写得最好的段落；
- trace 规则、d_min=16、2e+s<d_min 等技术口径全站一致；
- 攻击说明（attacks.py 文案）与案例 goal 描述准确且互相吻合。

## 三、前端怎么改：交互与视觉建议

按"投入产出比"排序。第 1~3 条值得做，后面的看时间。

1. **裁决卡不要"剧透"**。现在 `selectCase` 一选案例就立刻 `renderVerdict()`（app.js:234），
   评委往下一滚就直接看到结局，八步故事流的悬念全没了。建议：初始渲染为"封存"状态
   （模糊处理或只显示"⏳ 看完八步演示后揭晓"），当 `STEP` 第一次到达最后一步时再揭开。
   改动很小（renderStep 里加一个判断），叙事收益很大——这也正好呼应队长要的"演示感"。

2. **自动播放节奏按步给时长**。固定 2.6s（app.js:314）对第 1、5 步够用，对第 6 步
   （两栏探针全文）和第 7 步明显不够。建议给 timeline 每步配权重，如
   `[3, 3, 4, 4, 4, 6, 5, 6]` 秒，或者简单点统一调到 4s。

3. **≤640px 时导航消失，移动端会"迷路"**。styles.css:489 直接 `display:none` 掉
   nav-links，手机上从指标页回不到演示页（只能点 logo 回首页）。最低成本方案：
   640px 以下保留 nav-links 但缩小字号/间距；或者把三个视图链接收进一个简单的下拉。

4. **指标页是死胡同**。看完指标没有"回到演示试下一个案例"的出口。在指标页底部加一个
   反向 CTA（"← 回到演示，换一个攻击案例试试"），和 #6 的 verdict CTA 形成闭环。

5. **案例卡分组**。五张卡现在平铺，建议按买家分两行（Buyer 1 三张 / Buyer 2 两张），
   或按攻击类型排序（无攻击 → 第一类 → 第二类），让"难度递进"的结构可见。
   另外推荐给「buyer1_attack_type2（最难场景）」加一个"推荐演示"角标——评委时间有限时，
   这是最能体现方法价值的案例。

6. **SSE 逐步播放接口目前无人使用**（后端 `/stream` 写好了，前端没接）。两个选择：
   要么接上做成"实验运行中"的逐步揭示动画（演示效果很强，工作量中等）；
   要么本次比赛先不管它（接口留着不碍事）。不建议为它阻塞其他事。

7. 代码层面无需大动：无框架纯原生的选型对这个体量是对的；innerHTML 注入的文案全部来自
   自家后端，演示场景下 XSS 风险可接受；esc() 该用的地方都用了。

## 四、Homepage 设计：怎么理解"提出问题 → 预设场景 → 演示方法"

### 4.1 队长这个要求到底在说什么

这是经典的**叙事漏斗**（Apple 发布会、优秀论文 demo 站都是这个结构）：

| 段落 | 作用 | 回答评委心里的哪个问题 |
|------|------|----------------------|
| ① 抛问题 | 用一个评委能共情的问题制造钩子 | "为什么我要关心这件事？" |
| ② 设场景 | 把抽象问题落到一个**具体的、有主角的**故事里 | "这个问题长什么样？" |
| ③ 演示方法 | 所有技术步骤都变成"解决这个场景"的答案 | "你们是怎么解决的？真的有效吗？" |

关键洞察是：**②场景是①问题和③方法之间的桥**。没有场景，hero 三卡再漂亮也只是"陈述"；
有了场景，后面八步故事流里的每一步都有了主角（你、4 个买家、1 个可疑服务）和悬念
（到底是谁泄露的？）。评委不是在"看功能列表"，而是在"追一个案子"。

你们现在的 homepage 其实已经有 ①（h1 是问句）和 ③（流程胶囊 + CTA），**唯独缺 ②**——
这就是队长觉得"差点什么"的原因。《修改计划.md》第 8 条的方向判断是对的，下面给可直接落地的方案。

### 4.2 推荐结构：三屏叙事

**第一屏 · 抛问题**（强化现有 hero）

> **H1**：你花三个月打磨的 Skill，上架两周就出现了盗版。
> **H2/问句**：你能证明它是你的吗？能找出是谁泄露的吗？
> **lead**：Skill 是高价值数字资产，却"可见即可复制"。现有 prompt 级水印最多告诉你
> "确实被盗了"——但**追不到是哪个买家泄露的**。

（把现在"背景与痛点"卡的内容融进这一屏的文案，不再单独占一张卡。）

**第二屏 · 设场景**（新增，这是核心增量）

> 设定只用三句话讲完：
>
> 1. 你把 `code_review` Skill 卖给了 **4 个买家**。
> 2. 某天，市场上出现一个**可疑服务**，行为和你的 Skill 一模一样。
> 3. 你看不到它的内部——只能像普通用户一样，**黑盒**地向它提问。
>
> 悬念句（承上启下）：**只靠提问，我们能证明它是你的、并指认出泄露的买家。往下看 ↓**

配图建议：一个极简示意图——左侧"你（所有者）"，中间 4 个买家头像（buyer_1~4），
右侧一个打问号的"可疑服务"黑盒，买家中某一个到黑盒之间画一条虚线（先不揭示是谁）。
用现有的 `.hero-flow` 胶囊风格 + 几个 SVG 圆角矩形就能拼出来，不需要插画资源。
**这张图的"虚线不指明是谁"就是整个网站的悬念引擎。**

**第三屏 · 演示方法**（重组现有元素）

- 一行三卡：保留"核心创新"三个点（SkillIR / AGC 胶囊 / CV-ECC），每个一句话；
- "实验验证"卡改造成 **Apple 风格大数字条**（stat strip）：`99.2% 所有权判定 · 99.3% 买家溯源 ·
  1/15 水印开销`——大数字比四行 ✅ 列表冲击力强得多，也更省空间；
- 保留六节流程胶囊（加水印 → 卖给买家 → … → 追出买家）作为"方法预告片"；
- CTA 文案从"选择一个案例，开始演示 ↓"升级为场景的延续：
  **"进入演示：亲眼看完追查这名买家的 8 步 →"**。

### 4.3 为什么这样设计是"好的 homepage"

给队里一个可以对外讲的判断标准——好的演示首页满足三条：

1. **10 秒测试**：评委只看第一屏 10 秒，能复述出"这是解决什么问题的"（→ 抛问题）；
2. **有主角**：后续每个技术步骤都能回答"这一步在帮场景里的谁做什么"（→ 设场景）；
3. **钩子向下**：每一屏结尾都让人想往下滚（问题→"那怎么办？"，场景→"真能追到？"，
   方法→"点进去看"）。

### 4.4 落地拆解（与现有架构对齐）

与《修改计划.md》#8 的拆解一致，补充具体一点：

1. **文案**：`ui_text.py` 的 `hero` 重写 h1/lead，新增 `hero.scenario`
   （建议结构：`{title, steps: [三句话], suspense, diagram_labels: {...}}`），
   "实验验证"卡改为 `hero.stats: [{value:"99.2%", label:"所有权判定准确率"}, ...]`；
2. **结构**：index.html `#sec-intro` 内、`hero-grid` 与 `hero-flow` 之间插入
   `<div id="hero-scenario">`；
3. **样式**：styles.css 新增 `.hero-scenario`（沿用羊皮纸底 + 白卡 + 现有入场动画延迟序列，
   把 `.hero-inner > :nth-child(n)` 的 delay 链顺延一位即可）；
4. 示意图用内联 SVG 写死在渲染器里（图形不是文案，可以留在前端），文字标签走后端。

工作量评估：文案半天 + 前端一天以内。建议先把 4.2 的文案草稿拿给队长过目定调
（尤其"盗版上架"这类表述的风格强度），定稿后再动代码——顺序和修改计划的结论一致。
