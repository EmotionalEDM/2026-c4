## 1. Step8 卡片结尾字体大小不统一

**现象**：第 8 步（买家解码排名）结尾几行字号跳变——
"从真实输出还原的观测码字"、"把观测码字与每个买家的标准码字比对…"、"溯源结果…"这几句大小不一。

**根因**：渲染器 [app.js:614-653 `decode_grid`](frontend/app.js#L614-L653) 里这几行用了不同的类与内联字号：
- `observed_label` → `.bitlabel`（12px，灰）
- `rank_intro` → `.kv`（14px）
- `result_line` → `.kv` 但内联 `style="font-size:16px"`
- `ecc_line` → `.kv`（14px）

三种字号 + 一个内联覆盖，挤在卡片结尾，所以显得乱。

**修改方案**（统一为两级：小标题 12px + 正文 14px，结论行单独加重而非单纯放大）：
1. 在 [app.js:638-652](frontend/app.js#L638-L652) 把 `result_line` 那行的内联 `style="margin-top:14px;font-size:16px"` 改为 `class="kv result-line-strong"`（去掉内联字号）。
2. 在 styles.css 新增结论行样式，用字重/颜色而非字号制造层级：
   ```css
   .decode-summary{font-size:14px;font-weight:600;margin-top:14px}
   ```
3. 检查同位置 `observed_label`（`.bitlabel`）与 `rank_intro`（`.kv`）：把 `rank_intro` 也降为 `.bitlabel` 风格的小标题，保证"标签 12px / 数据正文 14px"两级一致。

**验收**：第 8 步从 bit 网格往下到结论行，字号只剩两档，视觉连贯。

## 2. ④ 结果裁决：信息层级反了（重点）

**现象**：以"所有权认定"一栏为例，现在"✅已确认"字号最大、盖过了标题"所有权认定"；指标行被挤在最下。组长要求的层级应是：
1. **「所有权认定」** —— 标题，最大；
2. `True-WS 1.0000 · False-WS … · Margin …` —— 指标行；
3. **「✅ 已确认」** —— 结论，字号**不得超过**标题。
右侧"买家溯源"一栏同理。

**根因**：
- 顺序问题——[app.js:685-694 `renderVerdict`](frontend/app.js#L685-L694) 当前 DOM 顺序是 `ownership_label → verdict-stat(已确认) → ownership_stats`，指标行在最后。
- 字号问题——`.verdict-stat` 是 `clamp(26px,3.4vw,36px)`（[styles.css:384](frontend/styles.css#L384)），而标题 `.kv` 只有 14px，所以结论比标题大得多。

**修改方案**：
1. **调整顺序**——在 [app.js:685-694](frontend/app.js#L685-L694) 两栏内部都改为「标题 → 指标 → 结论」：
   ```
   <div class="verdict-cap">所有权认定</div>      // 标题（原 ownership_label）
   <div class="kv">True-WS … · False-WS … · Margin …</div>   // 指标（原 ownership_stats，上移）
   <div class="verdict-stat ...">已确认</div>        // 结论（下移到最后）
   ```
   买家溯源栏（attribution_label / attribution_stats / verdict-stat）做同样调整。
2. **新增标题样式 + 压低结论字号**，在 styles.css：
   ```css
   .vcard.big .verdict-cap{font-size:22px;font-weight:600;color:var(--ink-on-dark);margin:0 0 8px}
   .vcard.big .verdict-stat{font-size:18px}   /* 结论 ≤ 标题 22px */
   ```
   （`.vcard.big` 是暗色大卡，已有 `.vcard.big .kv` 等深色覆盖，加这两条即可。）

**注意**：`.verdict-stat` 还被第 7 步评分卡等处复用，**不要**全局改它；只在 `.vcard.big` 作用域内覆盖，避免误伤别处。

**验收**：裁决大卡两栏都呈"标题最大 → 指标 → 结论（不超标题）"的层级。

## 3. 点击第一/二类攻击卡片直接跳到 Step5

**现象**：组长希望——点"第一类攻击 / 第二类攻击"的案例卡时，故事流默认从 **Step5（攻击者复制/改写）** 开始，因为前四步与 buyer_1 无攻击案例完全相同，评委不必重复看。

**根因**：[app.js:209-235 `selectCase`](frontend/app.js#L209-L235) 里固定 `STEP = 0`（第 217 行）。

**首选方案（组长 A 案）**：按攻击类型决定初始步：
- 在 `selectCase` 拿到 `CURRENT` 后，读 `CURRENT.attack` / 案例的 `attack` 字段判断；
- 若是 `type1_rewrite` 或 `type2_*`（非 `none`）→ 把 `STEP` 设为"攻击复制"那一步的下标。
  **不要硬编码 `STEP=4`**，应按 `phase` 或 step.index 查找，稳妥写法：
  ```js
  const isAttack = CURRENT.attack && CURRENT.attack.type && CURRENT.attack.type !== "none";
  const atkIdx = CURRENT.timeline.findIndex(s => s.index === 5);
  STEP = (isAttack && atkIdx >= 0) ? atkIdx : 0;
  ```
  （需先确认 timeline 各 step 的 `index` 字段命名——见下方"待确认"。）
- `buildStepper()` / `renderStep()` 已支持任意起始 STEP，前面 1~4 步仍可点进度条回看，符合组长意图。

**备选方案（组长 B 案）**：无攻击案例直接**跳过 Step5**。这需要后端在无攻击 timeline 里不产出第 5 步，或前端过滤——改动更大、且打乱"八步"叙事，**不推荐**，仅作兜底。

**建议**：采用 A 案（仅改 `selectCase` 一处，最小侵入）。

**待确认**：`CURRENT.attack` 的字段结构、以及 timeline 里"攻击复制"步是不是 `index === 5`。需读后端 [backend/app/cases.py](backend/app/cases.py) / [pipeline.py](backend/app/pipeline.py) 确认后再定下标查找条件。

## 4. Step1~8 进度条字体再大一点

**现象**：顶部 8 个步骤 pip 上的"STEP n / 标题"字太小。

**根因**：[styles.css:208-209](frontend/styles.css#L208-L209)
- `.pip-no` = 10.5px
- `.pip-tt` = 13px

**修改方案**：适度上调，并复核 pip 最小宽度避免换行：
```css
.pip-no{font-size:12px}      /* 10.5 → 12 */
.pip-tt{font-size:15px}      /* 13 → 15 */
```
若放大后窄屏换行难看，同步把 [styles.css:201](frontend/styles.css#L201) `.stepper .pip` 的 `min-width:96px` 略增到 `110px`。

**验收**：进度条文字清晰，且 8 个 pip 在 1080 容器内仍单行排布。

## 5. Step3 左边"门控规则"框大小不对

**现象**：第 3 步（所有者水印）左栏"新增到 Skill 的门控规则"框，相对右栏"五字段内部胶囊"显得比例失衡/内容被裁。

**根因**：[app.js:483-484](frontend/app.js#L483-L484) 左栏 `.rule-list` 内联 `max-height:300px;overflow:auto`，而右栏有表格 + 示例 `<pre>`，整体更高。`.cols2` 默认 `align-items:stretch` 会把两栏拉到等高，于是左栏出现"内框 300px 滚动、外框被撑高留白"的错位观感。

**修改方案**（二选一，推荐 A）：
- **A（推荐）**：让两栏顶端对齐、各自然高，不强行等高。在 styles.css 针对该步容器：
  ```css
  .step-visual .cols2{align-items:start}
  ```
  并把 [app.js:484](frontend/app.js#L484) 的 `max-height:300px` 调大到 `360px` 或与右栏 `<pre class="scroll">`（440px）协调，减少内外框落差。
- **B**：保持等高，但把左栏滚动区撑满卡片——用 flex 让 `.rule-list` `flex:1`。改动更多，非必要不做。

**验收**：第 3 步两栏视觉平衡，左框不再有突兀留白或过窄滚动。

## 6. "指标"页缺少其他入口

**现象**：`#metrics` 视图只能靠右上角导航"指标"进入；首页可点 logo / "演示"进入演示，但"指标"评委容易忽略。

**根因**：入口只有 [index.html:30 `#nav-links`](frontend/index.html#L30) 一处（由 [app.js:130](frontend/app.js#L130) 用 `UI.meta.nav` 渲染）。

**修改方案**（加一条显眼的"顺流"入口，不动导航）：
1. 在 ④ 结果裁决区底部加一个 CTA → `#metrics`。在 [index.html:119-123 `#sec-verdict`](frontend/index.html#L119-L123) 末尾加：
   ```html
   <a class="cta" id="verdict-cta" href="#metrics" hidden></a>
   ```
2. 文案进后端：在 [ui_text.py](backend/app/ui_text.py#L89) 的 `sections.verdict` 里加 `"cta": "查看整体实验指标 →"`。
3. 在 [app.js `renderVerdict`](frontend/app.js#L673) 末尾把该 CTA 显示出来（拿到裁决结果后 `removeAttribute("hidden")`），并在 `applyStaticText` 填文案。
4. （可选）首页 hero 三卡中"实验验证"卡也可加一条指向 `#metrics` 的小链接，给评委第二条路径。

**验收**：看完单案例裁决后，自然能一键进入指标页；无需依赖右上角导航。

## 8. homepage 重新设计（待定稿）

**组长要求**：homepage 要"**提出一个问题，预设一个场景，然后后面演示我们的方法**"。

**理解与建议方向**（供讨论，落地前先和组长对齐）：
当前 hero（[ui_text.py:50-72](backend/app/ui_text.py#L50-L72)）已有一句问句标题"当一份 Skill 被盗走，我们如何证明并追凶？"，方向是对的，但"问题→场景→方法"三段式还不够清晰。建议把首页重构成三屏叙事：

1. **抛问题**（保留并强化现有 h1）："Skill 被盗，你能证明它是你的、并指认是谁泄露的吗？"
2. **设场景**（新增一块）：用一句话讲一个具体故事——
   "你把 `code_review` Skill 卖给 4 个买家，其中一人改写后转售。黑盒下，只看输出，能否追到他？" 配 4 个买家 + 1 个泄露者的小示意图。
3. **演示方法**（现有"嵌入→验证→溯源"flow + CTA）：自然过渡到 ② 案例选择，点 CTA 进入演示。

**落地拆解**（定稿后再做）：
- 文案：改 `ui_text.py` 的 `hero`，新增 `hero.scenario` 字段承载"场景"段。
- 结构：[index.html:51-59 `#sec-intro`](frontend/index.html#L51-L59) 在 `hero-grid` 与 `hero-flow` 之间插入场景块。
- 样式：styles.css 新增 `.hero-scenario`，沿用现有卡片 / 入场动画节奏。

**当前结论**：第 8 条是开放设计题，**先和组长确认"场景"具体讲哪个案例、要不要配图**，再动手。建议作为第二批迭代，不阻塞 1~6 的样式/交互修复。
